# financialKG 前后端仓库分析报告

> 分析对象：`/home/ubuntu/code/financialKG`
>
> 结论：这是一个以 **Vue 3** 为前端、以 **Django + DRF** 为后端、以 **Neo4j + SQLite** 为数据底座的金融知识图谱系统。仓库已经具备“图谱管理 + 元知识管理”的基本能力，但当前实现存在明显的环境硬编码、前后端接口漂移、认证保护不足和部署配置老化问题。

---

## 1. 仓库总体定位

仓库由两个主要部分组成：

- `front/`：Vue CLI 前端应用，负责登录、节点管理、关系查询、元知识管理等页面。
- `knowledgegraph/`：Django 后端，负责认证、图谱节点/关系接口、元知识接口，以及对 Neo4j 和 SQLite 的读写。

核心调用链如下：

```text
浏览器
  -> Vue Router + axios
  -> Django URLs / Views
  -> Neo4j（企业图谱）
  -> SQLite（用户、元知识、公式、变量）
```

关键入口：

- 前端入口：`front/src/main.js:1`
- 前端路由：`front/src/router/index.js:1`
- 后端命令入口：`knowledgegraph/manage.py:7`
- 后端 URL 入口：`knowledgegraph/knowledgegraph/urls.py:25`
- Neo4j 连接初始化：`knowledgegraph/knowledgegraph/views.py:37`
- SQLite 配置：`knowledgegraph/knowledgegraph/settings.py:134`

## 3. 前端分析

### 3.1 技术栈

前端依赖定义在 `front/package.json:5`：

- Vue 3：`front/package.json:17`
- Vue Router 4：`front/package.json:18`
- Element Plus：`front/package.json:15`
- axios：`front/package.json:11`
- d3：`front/package.json:14`
- Vue CLI 构建脚本：`front/package.json:6`

### 3.2 应用入口与全局配置

`front/src/main.js:11` 到 `front/src/main.js:20` 负责：

- 从 `localStorage` 读取 token 并设置 axios 默认请求头
- 开启 `withCredentials`
- 设置 `axios.defaults.baseURL`
- 把 axios 挂载到全局属性

但这里有一个关键不一致：

- `main.js` 读取的是 `jwt_token`：`front/src/main.js:11`
- 登录页写入的是 `access_token` / `refresh_token`：`front/src/views/Login_neo4j.vue:72`
- 路由守卫检查的也是 `access_token`：`front/src/router/index.js:77`

结论：**刷新页面后，路由可能仍认为“已登录”，但 axios 默认 Authorization 头未必正确恢复。**

### 3.3 路由与页面

路由定义见 `front/src/router/index.js:12`：

- `/login`：登录页
- `/home`：首页
- `/addnode`：新增节点
- `/querynode`：查询节点
- `/deletenode`：删除节点
- `/queryrelationship`：关系查询
- `/allgraph`：图谱总览
- `/allmetaknowledge`：元知识列表
- `/addmetaknowledge`：新增元知识
- `/formula`：公式相关页面

其中只有 `/home` 显式标记了 `requireAuth`：`front/src/router/index.js:14` 到 `front/src/router/index.js:19`。

### 3.4 API 调用方式

当前前端**没有统一 API 封装层**，大量页面直接在组件里 `import axios` 后调用硬编码 URL。

典型例子：

- 登录：`front/src/views/Login_neo4j.vue:66`
- 查询节点：`front/src/views/QueryNode.vue:149`
- 删除节点：`front/src/views/DeleteNode.vue:225`
- 元知识列表：`front/src/views/AllMetaknowledge.vue:81`
- 模糊匹配：`front/src/views/Fuzzy_match.vue:71`

调用地址存在三类混用：

- `http://10.176.22.62:8001/...`
- `http://localhost:8001/...`
- `main.js` 中的生产 `baseURL = http://10.176.22.62:8000`：`front/src/main.js:17`

结论：**当前前端并不是“换一个 baseURL 就能跑”的结构，很多页面仍依赖写死的主机和端口。**

### 3.5 未接入或已漂移的前端页面

`front/src/views/` 下还存在一些页面文件，但没有接入当前路由：

- `CompareName.vue`
- `Fuzzy_match.vue`
- `SubGraph.vue`
- `AddNode_excel.vue`
- `AddRelationship_excel.vue`
- `QueryNode_excel.vue`

其中部分页面还依赖后端未开放的接口：

- `/comparename/` 在后端路由中被注释：`knowledgegraph/knowledgegraph/urls.py:35`
- `SubGraph.vue` 使用 `/generatequery/`，但当前后端 `urls.py` 中未定义该接口

这说明前端存在**历史页面遗留**和**接口漂移**现象。

- uWSGI：`knowledgegraph/requirements.txt:126`

### 4.2 配置与中间件

后端主配置文件为 `knowledgegraph/knowledgegraph/settings.py:1`。

关键点：

- `DEBUG = False`：`knowledgegraph/knowledgegraph/settings.py:66`
- `ALLOWED_HOSTS = ['*']`：`knowledgegraph/knowledgegraph/settings.py:68`

## 5. 前后端联调现状

从代码看，当前联调不是标准化环境配置驱动，而是依赖若干硬编码地址。

### 5.1 端口现状

- 前端开发服务器通常是 8080（Vue CLI 默认）
- 前端多个页面请求的是 `8001`
- `main.js` 的生产 `baseURL` 使用 `8000`：`front/src/main.js:17`
- uWSGI socket 又配置为 `8002`：`knowledgegraph/uwsgi.ini:3`

这说明端口规划并不统一。

### 5.2 对本地开发的影响

如果直接按常规方式启动：

- 前端：`npm run serve`
- 后端：`python manage.py runserver 0.0.0.0:8000`

那么至少会遇到这些问题：

1. 前端部分页面仍会访问 `8001`，不会走 8000。
2. 前端部分页面写死了 `10.176.22.62`，在其他机器上不可直接复用。
3. CORS 白名单未包含 `localhost:8080`，本地浏览器可能被拦截。

结论：**当前代码更接近“面向固定机器部署”的状态，不是开箱即用的本地开发结构。**

## 6. 主要问题与风险

### 6.1 环境与凭据硬编码

硬编码位置包括：

- Django `SECRET_KEY`：`knowledgegraph/knowledgegraph/settings.py:18`
- Neo4j 连接与认证：`knowledgegraph/knowledgegraph/views.py:37`
- 前端接口主机与端口：如 `front/src/views/Login_neo4j.vue:66`
- uWSGI 历史绝对路径：`knowledgegraph/uwsgi.ini:4`

影响：环境迁移困难，且安全边界弱。

### 6.2 认证状态不一致

- `main.js` 用 `jwt_token`：`front/src/main.js:11`
- 登录页与路由守卫用 `access_token`：`front/src/views/Login_neo4j.vue:72`、`front/src/router/index.js:77`

影响：登录态恢复和请求头行为不一致。

### 6.3 接口保护不足

- 全局默认权限是 `AllowAny`：`knowledgegraph/knowledgegraph/settings.py:95`
- 多个关键类视图直接 `permission_classes = []`：`knowledgegraph/knowledgegraph/views.py:228`、`knowledgegraph/knowledgegraph/views.py:263`、`knowledgegraph/knowledgegraph/views.py:297`、`knowledgegraph/knowledgegraph/views.py:331`

影响：当前“支持登录”不等于“所有敏感接口都被登录保护”。

### 6.4 Cypher 拼接风险

多个查询直接通过 f-string 拼接用户输入：

- `knowledgegraph/knowledgegraph/views.py:156`
- `knowledgegraph/knowledgegraph/views.py:172`
- `knowledgegraph/knowledgegraph/views.py:181`
- `knowledgegraph/knowledgegraph/views.py:198`

影响：存在 Cypher 注入风险，尤其是查询条件和关系名来自请求参数时。

### 6.5 前后端接口漂移

- 前端存在未接入页面
- 部分页面访问后端未暴露的接口
- 地址/端口不统一

影响：仓库可维护性较差，新接手者很难快速判断哪些页面和接口仍然有效。

### 6.6 初始化脚本过时

`knowledgegraph/knowledgegraph/management/commands/create_metaknowledge.py:25` 仍引用已删除字段。

影响：初始化文档如果误用该脚本，会直接报错。

### 6.7 仓库中包含构建产物与运行态数据

- `front/node_modules/`
- `front/dist/`
- `knowledgegraph/db.sqlite3`

影响：仓库体积偏大，环境状态与源码耦合较深。

## 7. 适合当前仓库的结论性判断

### 7.1 这是什么类型的项目

这是一个**已可运行、但偏“实验室/固定环境部署”风格**的前后端项目，而不是一个完成了环境抽象和标准化交付的通用工程模板。

### 7.2 当前最稳定的功能面

相对更清晰、实现也更集中的是两块：

1. **企业图谱管理**
   - 节点新增、删除、查询
   - 关系查询与 Excel 导入

2. **元知识管理**
   - 元知识、公式、变量的 ORM 管理

### 7.3 当前最明显的工程短板

- 配置未环境化
- 权限体系未真正落地到核心接口
- 前端接口调用没有统一封装
- 路由、页面、后端接口之间存在历史残留

---

## 8. 建议的梳理顺序

如果后续继续治理这个仓库，建议优先级如下：

1. **先统一配置来源**
   - 把 Neo4j、后端地址、SECRET_KEY、CORS、日志路径抽到环境变量

2. **再统一前端 API 层**
   - 去掉组件内分散的硬编码 URL
   - 统一 token 键名

3. **再补齐接口保护**
   - 把默认权限从 `AllowAny` 收紧
   - 为图谱写接口应用已有权限类/装饰器

4. **最后清理历史页面与失效接口**
   - 删除未接入页面
   - 或补齐缺失的后端接口与路由
